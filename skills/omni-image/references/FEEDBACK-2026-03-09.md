# Field Feedback: omni-image skill (2026-03-09)

> **Status: All items implemented (2026-03-09).** Archived to `references/`. See SKILL.md for the current state.

Session context: Heavy production use generating ~40 images for a bar service pitch site.
Used `generate` and `edit` extensively, `describe` not used this session.

## What Worked Well

1. **Edit mode with reference photos is the killer feature.** Feeding a product photo of the actual glassware into `edit` produced dramatically better results than `generate` alone. The AI preserved the exact glass shape (ribbed coupe, diamond-faceted rocks) while filling it with the described contents. The SKILL.md documents this but doesn't emphasize it enough.

2. **JSON output is clean.** Parsing results was straightforward. The structured output made it easy to chain commands programmatically.

3. **`--quality high` with `--size 1536x1024` produced genuinely high-fidelity results** for background images. The 1024x1024 square format worked well for product/drink shots.

4. **Timeout tolerance is reasonable.** Some calls took 60-90 seconds, which is expected for high-quality edits.

## Issues / Improvement Opportunities

### 1. Missing `--input-fidelity` in edit examples (P1)
The `edit` section shows `--input-fidelity high` but doesn't explain when to use it vs. not. In practice:
- **Use `high`** when you want to preserve the structural layout of the source image (e.g., stylizing a venue photo while keeping walls/windows intact)
- **Leave it off** when using the source as a loose reference (e.g., "fill this glass with a different liquid")

Suggest adding a "When to use input fidelity" callout.

### 2. No guidance on "edit as structural reference" pattern (P1)
The most powerful pattern I discovered was: **use a product photo as input to `edit`, not to modify the product, but to use its shape as a structural anchor while completely reimagining the contents.** Example: feed a photo of an empty coupe glass and ask it to fill it with a specific blue cocktail. The SKILL.md doesn't mention this workflow at all.

Suggest adding a "Reference-Based Generation" section with examples like:
```bash
# Use a glass photo as structural reference, reimagine the contents
python3 scripts/omni_image.py edit \
  "Fill this coupe glass with bright blue butterfly pea tea cocktail" \
  -i ref_empty_coupe.jpg --quality high -o cocktail.png
```

### 3. No retry/resilience guidance (P2)
I hit `Connection error` / `APIError` on approximately 20% of calls. The script exits with code 1 and no retry. Three options:
- **Built-in retry**: Add `--retries N` flag with exponential backoff
- **Documentation**: Add a "Retries" section to Troubleshooting explaining the transient nature of these errors
- **Wrapper script**: Provide a `batch_generate.sh` helper that retries failed items from a manifest

### 4. No batch/manifest mode (P2)
When generating 8 drink images with consistent style, I had to fire 8 separate commands. A manifest-driven batch mode would be valuable:
```yaml
# drinks.yaml
style_prefix: "Vivid, hyper-real 1920s cinematic movie still."
defaults:
  quality: high
  size: 1024x1024
jobs:
  - name: cold_open
    input: ref_coupe.jpg
    prompt: "Blue butterfly pea tea cocktail with violet-shifting ice cubes..."
    output: drink_cold_open.png
  - name: smoking_gun
    input: ref_rocks.jpg
    prompt: "Golden amber with smoke cloche..."
    output: drink_smoking_gun.png
```
```bash
python3 scripts/omni_image.py batch drinks.yaml --output-dir ./public/images/
```

### 5. No `--prefix` or `--style-prefix` flag for consistent prompting (P3)
When generating a cohesive set of images, every prompt starts with the same style preamble ("Vivid, hyper-real 1920s cinematic movie still..."). A `--prefix` flag would reduce repetition and enforce consistency:
```bash
python3 scripts/omni_image.py generate "blue cocktail in a coupe" \
  --prefix "Vivid, hyper-real 1920s cinematic movie still. Rich jewel tones, warm golden lighting, film grain." \
  --quality high -o cocktail.png
```

### 6. Silent failure on empty output (P3)
When `edit` returned exit code 0 but no output text (happened once), there was no way to tell if the image was actually written. Adding a `--verify` flag that checks the output file exists and has nonzero size would add confidence.

### 7. Model name clarity (P3)
The doc says `gpt-image-1.5` is "newest and recommended" but doesn't mention what image resolution it defaults to at each quality level. Adding a table mapping `quality` x `model` to typical output resolution would help with planning.
